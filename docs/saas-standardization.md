# Li Xi Station SaaS Standardization

## Objective

Li Xi Station is moving from a Tet li xi helper into a SaaS prize-draw platform. The Tet experience remains the default campaign skin, but the system should support branded prize campaigns, uploaded assets, public draw links, billing, analytics, and reusable prize mechanics.

## Current State

- Frontend has been migrated to TanStack Start + Vite routes.
- Convex Auth schema/http/provider setup is present with Google OAuth, while legacy username/PIN login remains only as an env-gated backend migration/account-linking bridge.
- The `/auth` route is Google OAuth-only; legacy username/PIN account auth is no longer exposed in the host UI and backend legacy mutations stay server-disabled unless migration env flags are explicitly enabled.
- Auth account reads expose the current Convex Auth user only; arbitrary user-id profile lookup is not part of the public Convex API surface.
- Draw sessions support both kiosk mode and public claim links via `deliveryMode`; only link-mode sessions mint a `publicCode`.
- Public claim codes are normalized opaque 24-character hex tokens generated from 12 random bytes. Link creation re-checks the pending public-code bucket after insert before returning a share link, and guest reads/redeems only resolve when exactly one pending link-mode session matches the token, so duplicate legacy rows cannot ambiguously redeem a prize.
- Public claim links carry `publicCodeExpiresAt`, defaulting to 7 days after creation. Claim lookup reads pending rows through the `publicCode + status` index, then still requires exactly one non-expired link-mode session. Expired pending links no longer resolve for guests, no longer count toward open-session plan usage, and no longer lock budget reconfiguration. Unsafe or malformed expiry timestamps fail closed instead of leaving a claim open; explicit expiry values also fail closed if they exceed the 7-day TTL from the row's valid `createdAt`, and synthesized default expiry fails closed if `createdAt + TTL` would exceed the safe integer timestamp range, so imported rows cannot extend public claims indefinitely. Host cancellation also refuses expired or malformed link sessions instead of mutating them into normal cancelled rows. `migrations:cleanupExpiredPublicLinks` is the owner-scoped maintenance path for cancelling expired or malformed pending public links after a dry run while preserving audit fields. Legacy rows with `publicCode` but no `deliveryMode` are treated as link-mode rows, and migration backfill sets missing expiry only when `getPublicLinkExpiresAt` can derive a safe value; malformed creation timestamps or overflowed synthesized expiry values are reported through `skippedInvalidPublicCodeExpiry` instead of crashing the owner migration.
- Redeem access is mode-scoped: public link redemption must present the matching `publicCode`, while station redemption/cancel must resolve the owner through Convex Auth. Host-facing draw APIs do not accept the temporary legacy bridge.
- Host share-link controls open the public `/claim/<publicCode>` route instead of reusing the station guest screen, so link-mode sessions always redeem through the public mutation.
- Host draw state exposes recent pending link-mode sessions separately from the live station session, allowing hosts to copy, open, or cancel public links after refresh while preserving station-mode isolation. The live station session is only exposed when its campaign still exists, belongs to the host, and is active; malformed legacy station sessions with missing, foreign, or inactive campaigns stay out of the guest stage just as redemption fails closed.
- Public claim redemption uses `publicCode` only and returns only the prize outcome; internal Convex draw session ids and host budget state stay out of the guest-facing API contract.
- Public claim session reads expose only guest display name, public campaign copy/theme/media, expiry timestamp, and cosmetic prize kinds; internal campaign/session ids, public codes, host display identity, and host budget state stay out of the guest-facing API contract. Guest-facing branding comes from campaign snapshots or owner-checked campaign fallbacks, not raw Convex Auth/user fields or host profile display names.
- Public claim session reads fail closed when the session's scoped budget is missing, exhausted, or has no available prize units payable by the remaining budget, so guests do not enter the envelope animation for an unpayable legacy or malformed link.
- Public claim mutable campaign fallbacks are owner-checked before copy, theme, hero, or prize preview resolution, so malformed legacy campaign references cannot expose another host's campaign state.
- Campaign settings now exist in Convex and `/campaigns`, with multi-campaign selection, new draft creation, brand copy, public claim headline/subtitle/start CTA/collect CTA/waiting copy, theme, status, hero asset, and R2 upload wiring. Draw and public claim routes resolve the draw template from the campaign theme, so the initial `brand` template is a supported red/gold campaign variant driven by campaign copy and uploaded hero assets rather than an admin-only label.
- The local contract suite includes `scripts/test-saas-workflow-policy.mjs`, which guards the OAuth completion route, Campaign Studio decomposition, campaign-scoped budget setup, station/public-link draw creation, public claim redemption, entitlement callsites, and the live evidence fields required before production acceptance. Full browser/database E2E still belongs in the staged production evidence pass because Google OAuth, R2, and Polar require deployed credentials.
- Campaign workspace APIs reserve `activeCampaign` for an actual `active` campaign. Visible campaign reads use active/draft status buckets instead of scanning archived campaigns. Active campaign resolution is centralized and deterministic: it keeps the host profile's default campaign when that row is still active for the owner, otherwise it chooses the most recently updated active campaign. Campaign Studio may still select a draft for editing, but backend responses do not label drafts as active fallbacks. If a client supplies a `selectedCampaignId` outside the owner's visible workspace, the query fails closed instead of falling back to active campaign data or recent assets.
- Campaign deactivation is lifecycle-guarded: an active campaign cannot be moved to draft or archived while it still has open station or public-link sessions. Activating a different campaign is treated as an implicit deactivation of the current active campaign, so it is also blocked while the current active campaign has open sessions. Hosts must redeem/cancel pending sessions before hiding, pausing, or replacing a live campaign from Studio. The guard reads pending sessions through campaign+owner+status+delivery buckets before applying link-expiry open-state checks. Public claim lookup and redemption also fail closed if the linked campaign is missing, foreign-owned, or no longer active; the shared redemption path enforces the same campaign guard before budget mutation or analytics writes.
- Campaign slugs are unique per owner. Explicit saves reject duplicate slugs, and automatic default campaign creation uses a shared unique slug helper across setup, draw fallback, and Campaign Studio. Automatic default campaign slug candidates, including owner-id fallback candidates, must pass `by_owner_slug` lookup before being returned.
- Campaign assets now have a shared upload policy: hero images are limited to JPG, PNG, WebP, GIF, or AVIF, capped at 8 MB, and require campaign-scoped declared metadata before the backend mints an R2 signed upload URL. Upload URL minting requires a configured `R2_BUCKET`, stores that configured bucket on the `reserved` asset row, and later upload callbacks, metadata sync, attach, renderability, and ops readiness all use the same safe bucket-name policy: 3-63 lowercase DNS-style characters, no adjacent dot/dash separators, and no IPv4-literal bucket names. All asset paths fail closed if the asset bucket differs from the current configuration. R2 upload and metadata callbacks canonicalize the object key with the same R2 key policy before querying asset rows, patching uploaded keys, or reading metadata, so raw callback/client key spelling cannot drift from the stored object identity. That reservation creates a quota-counted `reserved` asset row; the R2 upload callback only accepts `reserved` or already-`uploaded` rows, rejects duplicate owner/key rows as ambiguous, clears any campaign hero pointing at rejected rows, metadata sync only accepts `reserved`, `uploaded`, or `attached` rows, then attach rejects duplicate owner/key rows with the same reject-and-delete path, only promotes `reserved` or `uploaded` rows whose `campaignId` exactly matches the campaign that declared them, remains idempotent for already-`attached` renderable assets in that same campaign, and rejects `rejected`, malformed lifecycle rows, or legacy rows missing campaign linkage. Backend-detected rejected assets schedule R2 object deletion and stamp `r2ObjectDeleteScheduledAt` / `r2ObjectDeleteReason`, so invalid uploads, duplicate key rows, bad metadata, and stale abandoned reservations do not become unbounded storage retention. Attached assets may move to `attached` only after R2 metadata confirms the same policy. Campaign saves only accept renderable hero assets whose `campaignId` matches the campaign being saved, and Campaign Studio only lets hosts pick recent assets for the currently selected campaign. Asset quota usage counts only `reserved`, `uploaded`, and `attached` rows through the `ownerId + status` index, so rejected uploads or malformed legacy rows do not permanently consume plan asset quota. `migrations:cleanupStaleReservedAssets` is the dry-runnable maintenance path for rejecting abandoned R2 upload reservations so incomplete browser uploads do not hold asset quota forever. Campaign Studio keeps the uploaded R2 key available for retry when metadata is not yet readable, avoiding duplicate file selection.
- Campaign hero image URLs and Campaign Studio recent-asset previews are only generated from renderable `campaignAssets` owned by the campaign owner and scoped to the campaign being displayed: assets must be `attached`, belong to the configured R2 bucket, validated from R2 metadata, carry `validatedAt`, have policy-allowed image content type, and have a safe positive size no larger than the 8 MB upload cap. Rejected assets, raw `uploaded` assets, oversized assets, wrong-bucket assets, and legacy rows missing validation do not mint signed URLs. Campaign Studio recent assets are read from the selected campaign through the `campaignId + ownerId + status + createdAt` index instead of owner-wide asset history. Key-based signed URL reads require a campaign id, verify that campaign belongs to the owner, use the `key + ownerId` index, require exactly one owned asset row, and require that asset's `campaignId` to match the requested campaign so duplicate imported/legacy rows or cross-campaign keys do not resolve by first-match behavior. Internal render URL helpers also require a campaign id, re-check that the campaign belongs to the owner, and Campaign Studio uses that same centralized helper rather than minting direct R2 URLs. The public `assets:getAssetUrl` query is a thin owner/campaign wrapper around that same helper, so `convex/assets.ts` has only one backend callsite that mints R2 signed URLs. R2 upload callbacks and metadata sync also resolve by the authenticated owner plus key before mutating rows; both callback paths reject duplicate key rows for that owner and clear any campaign hero pointing at them instead of promoting ambiguous legacy/imported asset rows.
- Budgets are inventory-based (`amount x quantity`) and redemptions decrement one budget item.
- Convex Components are registered for Aggregate, Sharded Counter, R2, and Polar. Analytics, assets, and billing modules exist as integration surfaces.
- Owner authorization is centralized in `convex/authorization.ts`, with Convex Auth enforced for current host APIs. `requireResolvedOwner` no longer exposes a legacy owner bridge opt-in path; migration maintenance uses an authenticated-owner or migration-token resolver instead. Verified-owner helpers require a live Convex Auth identity and fail closed when unauthenticated so new SaaS APIs do not accidentally accept raw owner IDs.

- Host screens omit client-supplied owner identity for owner-scoped APIs. Host route components pass empty args only after `useOwnerSession` returns a live Convex Auth owner; the old `ownerBridgeArgs` helper and client owner-id arg type have been removed.
- The frontend host session hook now derives OAuth identity from Convex Auth `getCurrentUser` only. The `/auth` OAuth completion path clears stale legacy entries, materializes the authenticated host profile through `ensureCurrentHostProfile`, and then routes hosts that still need budget/PIN setup to `/setup`; already configured hosts land in Campaign Studio (`/campaigns`) before creating draw sessions. Stale localStorage legacy owner sessions are read only for cleanup, the frontend exports no local owner-session writer, and stale entries do not authorize host routes.

- Campaign open sessions are bounded by cumulative payable prize capacity: before minting another station or public-link session, the backend counts open pending sessions for that campaign through campaign+owner+status+delivery buckets and rejects creation when they already cover all remaining active prize units the current budget can actually pay for. Public claim previews, station pending-session reward pools, and redemption selection all use the same capacity-preserving prize filter, so guests are not shown prize types that would have to be rejected to keep enough payable units for the campaign's other open sessions. Expired links and cancelled/redeemed sessions do not reserve units.
- Draw session creation only accepts active campaigns. Draft campaigns can be configured in Campaign Studio, but they cannot mint station sessions or public claim links until activated.
- If a host already has visible draft campaigns but no active campaign, draw creation fails with an activation prompt instead of auto-creating another default campaign. Auto-default creation is reserved for first-run owners with no visible active/draft campaigns.
- Public claim sessions expose only unique available prize kinds that are payable by the remaining budget for cosmetic envelope animation; exact remaining prize inventory stays host-only.
- Draw sessions and redemptions now snapshot host/campaign public context: host display name/slug, campaign name/brand/description/theme, claim copy including separate start and collect CTAs, hero asset key, delivery mode, and public code. Historical results no longer depend only on mutable campaign records.
- Station guest mode and public claim reads treat a session campaign snapshot as authoritative: once a session has snapshot metadata, missing optional claim copy or hero key remains missing instead of falling through to the campaign's later mutable values. Mutable campaign fallback is reserved for legacy sessions that do not have snapshot metadata. Station guest mode also renders from the session-scoped capacity-preserving reward pool, so changing the active campaign while a guest is waiting cannot show the wrong brand, hero, copy, or prize inventory.
- `migrations:backfillOwnerSaaSModel` can promote one owner at a time into the standardized model, with `dryRun` support, skipped counts for ambiguous campaign state or foreign campaign/session references, and a bounded `auditDecisions` sample for rows requiring manual repair. Session backfill now reports foreign `campaignId` references and does not use them as snapshot targets; aggregate skipped counts combine both session and redemption foreign campaign references.

## Target Backend Model

- `users`: Convex Auth users with optional legacy host fields during migration.
- Host profile: operational fields such as globally unique display slug, onboarding state, and default campaign settings. Host PIN remains on `users` until the legacy account bridge is removed.
- Campaign: generalized prize program with owner-scoped unique slugs. Tet li xi is one campaign theme, not the only product model.
- Prize pool items: campaign-scoped configurable prize inventory, rarity, display metadata, and availability.
- Draw sessions: claimable sessions with public token/link, delivery mode, lifecycle status, campaign-scoped guest metadata, and redemption linkage.
- Redemptions: immutable prize outcome record with delivery mode and host/campaign snapshots.
- Assets: uploaded campaign media backed by Cloudflare R2, with entitlement gates before signed upload URL minting, metadata validation, rejected-asset filtering, strict owner/key resolution, owner-scoped callback mutation, and owner/campaign-scoped signed URL reads.
- Billing: Polar subscription state and plan limits.
- Analytics: aggregate totals plus sharded counters for high-write events, with owner-wide and campaign-scoped leaderboard/history reads.

## Backfill Command

Run a dry run before mutating legacy data:

```bash
npx convex env set LI_XI_MIGRATION_TOKEN <random-secret>
npx convex run migrations:backfillOwnerSaaSModel '{ "ownerId": "<user-id>", "dryRun": true, "migrationToken": "<random-secret>" }'
npx convex run migrations:backfillOwnerSaaSModel '{ "ownerId": "<user-id>", "migrationToken": "<random-secret>" }'
```

The backfill is conservative. It creates or updates host profile state, assigns missing draw `deliveryMode`, links legacy sessions/redemptions/budget rows only when a single target campaign can be resolved, fills missing host/campaign snapshots, and uses idempotent Aggregate inserts for redemption totals. If an owner has multiple visible active/draft campaigns and no default campaign, the migration reports `skippedAmbiguousCampaign` instead of guessing. Existing campaign/session references are used only when they belong to the owner being migrated; otherwise they are reported through skipped foreign-reference counters and excluded from campaign aggregate backfill. Session rows with foreign campaign references are also excluded from campaign snapshot target resolution so migration output exposes the bad reference instead of silently treating it as a no-campaign legacy row. Link-mode rows with malformed `createdAt` or unsafe synthesized expiry overflow are counted in `skippedInvalidPublicCodeExpiry`; they remain fail-closed at runtime until data is repaired manually. The response includes up to 50 `auditDecisions` samples for foreign campaign references, foreign session references, and invalid public-code expiry rows so operators can identify source records without returning an unbounded row list.

The mutation also accepts the current authenticated Convex Auth user without a token. CLI owner-ID backfills require `LI_XI_MIGRATION_TOKEN` / `LIXI_MIGRATION_TOKEN` and an explicit `ownerId`; migration maintenance resolves that owner through a token-aware migration resolver instead of the legacy owner bridge, so the public migration endpoint does not widen the bridge.
`LIXI_MIGRATION_TOKEN` is accepted as a compatibility alias; migration access rejects weak configured token values before comparing the caller input, compares accepted values through the shared timing-safe helper, and production readiness fails if either token env remains configured after the migration window.

Expired public claim cleanup uses the same authenticated-or-migration-token boundary:

```bash
npx convex run migrations:cleanupExpiredPublicLinks '{ "ownerId": "<user-id>", "dryRun": true, "migrationToken": "<random-secret>" }'
npx convex run migrations:cleanupExpiredPublicLinks '{ "ownerId": "<user-id>", "migrationToken": "<random-secret>" }'
```

It reads the oldest pending link-mode rows first, including legacy pending rows with missing `deliveryMode`, through owner/status/delivery buckets and only patches rows that the public-link expiry helper treats as expired or malformed. Dry-run output includes redacted session-level keep/cancel decisions with session id, delivery bucket, public-code presence, campaign id, timestamps, resolved expiry, and reason, but not the raw public claim code. Cleanup sets `status: "cancelled"` and `cancelledAt`; it does not delete public codes, snapshots, campaign references, or guest audit fields. `crons:cleanup expired public claim links` also runs the same cleanup hourly through the global `status + deliveryMode + createdAt` index so expired public links do not rely only on manual owner-scoped maintenance.

Host profile default campaign repair follows the same boundary:

```bash
npx convex run migrations:repairHostProfileDefaultCampaign '{ "ownerId": "<user-id>", "dryRun": true, "migrationToken": "<random-secret>" }'
npx convex run migrations:repairHostProfileDefaultCampaign '{ "ownerId": "<user-id>", "migrationToken": "<random-secret>" }'
```

It leaves valid active/draft defaults untouched, replaces archived/missing/foreign defaults with the latest active campaign or latest draft for the owner, and clears `defaultCampaignId` when no visible campaign remains. Dry-run output includes a profile decision with profile id, previous default, replacement default, action, and reason so operators can audit whether the row is missing, foreign, archived, already valid, or has no default.

Duplicate host profile repair follows the same boundary:

```bash
npx convex run migrations:repairDuplicateHostProfiles '{ "ownerId": "<user-id>", "dryRun": true, "migrationToken": "<random-secret>" }'
npx convex run migrations:repairDuplicateHostProfiles '{ "ownerId": "<user-id>", "migrationToken": "<random-secret>" }'
```

It scans one owner through the host profile `by_owner` index and returns
profile-level decisions with slug, default campaign, onboarding state, and
timestamps so the dry-run can be audited. It keeps the most recently updated
profile and deletes extra rows only on apply so host identity, public slug, and
default campaign resolution can return to a unique-owner profile invariant.

Active campaign repair follows the same boundary for legacy/imported owners with
more than one `active` campaign:

```bash
npx convex run migrations:repairOwnerActiveCampaigns '{ "ownerId": "<user-id>", "dryRun": true, "migrationToken": "<random-secret>" }'
npx convex run migrations:repairOwnerActiveCampaigns '{ "ownerId": "<user-id>", "migrationToken": "<random-secret>" }'
```

It keeps the active campaign already selected as the host default when possible,
otherwise keeps the most recently updated active campaign. Dry-run output includes
campaign-level keep/demote decisions with default-selection and timestamp metadata,
plus a profile decision showing the previous and next default campaign. Extra
active rows are demoted to `draft`, and an existing host profile is pointed at the kept active
campaign so active-campaign reads no longer depend on arbitrary index order.

Duplicate budget repair follows the same boundary for imported owners that have
more than one `ownerBudgets` row in the same owner/campaign scope:

```bash
npx convex run migrations:repairDuplicateOwnerBudgets '{ "ownerId": "<user-id>", "dryRun": true, "migrationToken": "<random-secret>" }'
npx convex run migrations:repairDuplicateOwnerBudgets '{ "ownerId": "<user-id>", "migrationToken": "<random-secret>" }'
```

It scans budgets for one owner through the `by_owner` index, groups by
`campaignId` including the legacy missing-campaign scope, returns budget-level
keep/delete decisions with scope, totals, remaining budget, and timestamps for
dry-run audit, keeps the most recently updated row, and deletes extra rows so
setup and payout can return to the unique budget-scope invariant.

Stale R2 upload reservation cleanup also follows the same boundary:

```bash
npx convex run migrations:cleanupStaleReservedAssets '{ "ownerId": "<user-id>", "dryRun": true, "migrationToken": "<random-secret>" }'
npx convex run migrations:cleanupStaleReservedAssets '{ "ownerId": "<user-id>", "migrationToken": "<random-secret>" }'
```

It reads the oldest `reserved` assets first through the owner/status index, returns asset-level keep/reject decisions with key, campaign, bucket, metadata, age, and rejection reason for dry-run audit, rejects only reservations older than the configured age window, and leaves uploaded/attached assets untouched. The default window is 24 hours, with `olderThanMs` clamped between 1 hour and 30 days. `crons:cleanup stale campaign asset reservations` also runs hourly through the global `status + createdAt` index so abandoned reservations do not rely only on manual owner-scoped maintenance.

Analytics backfills follow the same operational boundary. Authenticated hosts can backfill their own analytics from the app or CLI session; unauthenticated CLI runs must pass `LI_XI_MIGRATION_TOKEN` / `LIXI_MIGRATION_TOKEN` and an explicit `ownerId`. Owner backfill repairs owner Aggregate redemption totals plus owner Sharded Counter session/redemption events, and also repairs campaign Aggregate/counter state for redemption rows whose `campaignId` belongs to the owner. Campaign backfill updates campaign Aggregate redemption totals and campaign Sharded Counter session/redemption events. Campaign backfill uses campaign+owner indexes for scoped rows and only scans owner history for legacy unscoped redemptions that may need a campaign patch. When a previously owner-only event marker is later linked to a campaign, the marker is patched and only the campaign counter is incremented. Analytics dry-runs scan rows and report `...WouldBackfill` / `redemptionsWouldPatchCampaign` estimates, but do not write Aggregate rows, Sharded Counter event markers, counter increments, or legacy redemption campaign patches, so write counters remain zero. `...CounterEventsWouldBackfill` estimates marker inserts/patches, while `...CounterIncrementsWouldBackfill` estimates Sharded Counter increments. Both paths use `analyticsCounterEvents` markers so reruns do not double-count:

```bash
npx convex run analytics:backfillOwnerRedemptionAggregate '{ "ownerId": "<user-id>", "dryRun": true, "migrationToken": "<random-secret>" }'
npx convex run analytics:backfillOwnerRedemptionAggregate '{ "ownerId": "<user-id>", "migrationToken": "<random-secret>" }'
npx convex run analytics:backfillCampaignAnalytics '{ "ownerId": "<user-id>", "campaignId": "<campaign-id>", "dryRun": true, "migrationToken": "<random-secret>" }'
npx convex run analytics:backfillCampaignAnalytics '{ "ownerId": "<user-id>", "campaignId": "<campaign-id>", "migrationToken": "<random-secret>" }'
```

## Legacy Auth Bridge

Google OAuth is the production account path. The old username/PIN account form is no longer shown in the host UI. Frontend route guards require Convex Auth browser storage, `useOwnerSession` derives identity from the live Convex Auth user, stale legacy localStorage entries are cleared/ignored, and host screens call owner-scoped APIs without client-supplied `ownerId`. Production setup, host-profile, plan-state, analytics reporting, asset URL, and leaderboard/history APIs reject the legacy bridge and require Convex Auth. The backend legacy login/register mutations remain only as an env-gated migration/account-linking bridge and reject calls unless both backend flags are set: `LI_XI_ENABLE_LEGACY_AUTH=true` / `LEGACY_AUTH_ENABLED=true` and `LI_XI_ENABLE_LEGACY_OWNER_BRIDGE=true` / `LEGACY_OWNER_BRIDGE_ENABLED=true`. `requireResolvedOwner` has no legacy bridge opt-in; migration maintenance uses authenticated owner access or `LI_XI_MIGRATION_TOKEN` / `LIXI_MIGRATION_TOKEN` plus an explicit `ownerId`. Production readiness fails if either backend or frontend legacy auth/bridge flags remain enabled. OAuth completion now creates or updates the host profile before host routing, so Google users enter the SaaS host model immediately. When legacy login is temporarily enabled with the owner bridge for migration/account-linking, the backend still promotes the account into `hostProfiles`, normalizes missing legacy timestamps, and returns the host-profile display name so migration traffic uses the same host model.

Host PIN remains part of the product, but its role is operational confirmation for live draw/setup actions, not account authentication.
